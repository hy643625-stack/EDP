# 查找与哈希表

## 学习目标
- 能实现并分析顺序查找和二分查找的复杂度
- 能解释哈希表的工作原理（哈希函数、冲突处理）
- 能对比链地址法和开放地址法的优劣
- 能分析哈希表查找的平均复杂度和最坏复杂度

## 知识地图
```
查找技术
├── 顺序查找（无序数据）
│   ├── O(n) 时间、O(1) 空间
│   └── 适用：小数据量或偶尔查找
├── 二分查找（有序数据）
│   ├── O(log n) 时间、O(1) 空间
│   ├── 前提：数据有序
│   └── 变体：bisect_left / bisect_right
└── 哈希表
    ├── 哈希函数（均匀分布、确定性）
    ├── 冲突处理
    │   ├── 链地址法（链表/红黑树）
    │   └── 开放地址法（线性/二次/双重探测）
    ├── 动态扩容（负载因子触发 rehash）
    └── 复杂度：平均 O(1)、最坏 O(n)
```

## 核心讲解

### 二分查找的边界问题
二分查找看似简单，但边界条件容易出错：

```python
def binary_search(arr, target):
    left, right = 0, len(arr) - 1
    while left <= right:          # 注意：<= 而非 <
        mid = left + (right - left) // 2  # 防溢出
        if arr[mid] == target:
            return mid
        elif arr[mid] < target:
            left = mid + 1
        else:
            right = mid - 1
    return -1
```

### 哈希冲突处理对比

| 方法 | 优点 | 缺点 | 典型实现 |
|------|------|------|---------|
| 链地址法 | 实现简单、删除方便、对高负载容忍好 | 指针开销、缓存不友好 | Java HashMap |
| 线性探测 | 缓存友好、无指针开销 | 主聚集（primary clustering） | C++ unordered_map |
| 双重哈希 | 分布均匀、减少聚集 | 需计算两次哈希 | Python dict（变体） |

### 动态扩容
当 `负载因子 = 元素数 / 表大小` 超过阈值时，需要扩容并 rehash：
- 分配更大数组（通常是原大小的 2 倍）
- 将所有元素重新插入新表
- 均摊视角下，插入仍为 O(1)

## 典型例子
1. **两数之和**（LeetCode #1）：哈希表存储已遍历元素的值和索引
2. **LRU 缓存**（LeetCode #146）：哈希表 + 双向链表
3. **查找缺失元素**：哈希表 vs 排序比较的典型决策
4. **Python dict 内部实现**：紧凑型开放地址 + 伪随机探测

## 易错点
- 未负载因子过高前不扩容，导致查找退化为 O(n)
- 开放地址法的"懒惰删除"标志未正确处理
- 哈希表无法按序遍历（如需有序遍历，用 OrderedDict 或 TreeMap）
- 忘记可变对象作键会导致哈希值变化后的"幽灵键"

## 与后续模块的关系
哈希表是实现图的邻接表快速查找、动态规划的记忆化搜索的底层工具。理解哈希表对掌握 Python 的 dict/set 性能调优至关重要。
