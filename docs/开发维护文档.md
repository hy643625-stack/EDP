# 开发维护文档

本文档面向后续继续迭代 `EveryDayPerfect` 的开发者，重点记录当前工作区结构、日常开发流程、缓存清理方式和 Windows 环境下的注意事项。

如果你是第一次接手这个项目，先看：

- `docs/开发交接手册.md`

## 1. 当前工作区结构

桌面工作区固定分为四层：

- `01-source`：源码开发区
- `02-user-software`：面向用户的软件目录
- `03-send-package`：对外发送的压缩包
- `04-product-docs`：产品资料、方案文档、报告

主源码工程路径：

```text
C:\Users\Lenovo\Desktop\EveryDayPerfect\01-source\EveryDayPerfect
```

## 2. 为什么目录名保留 ASCII

当前顶层目录继续使用 `01-source`、`02-user-software` 这类 ASCII 名称，不改回中文，原因如下：

- PowerShell 在部分中文路径下容易出现编码或脚本解析问题
- 一些压缩包工具和安装脚本对中文路径兼容性较差
- Windows 上的虚拟环境、打包工具和前端工具链更容易在纯 ASCII 路径下保持稳定

结论：

- 目录名保持 ASCII
- 给人看的文档、说明、发布资料统一用简体中文

## 3. 日常开发流程

### 3.0 一页式流程

如果只想记最短路径，按这个顺序：

1. 在 `01-source\EveryDayPerfect` 改代码
2. 执行 `.\start_everydayperfect.ps1`
3. 本地验证修改
4. 执行 `.\set_project_version.ps1`
5. 更新 `CHANGELOG.md`
6. 跑测试
7. 需要时执行 `.\clean_local_workspace.ps1`
8. 发版时执行 `.\publish_windows_release.ps1`

### 3.1 启动项目

在源码工程根目录执行：

```powershell
Set-ExecutionPolicy -Scope Process -ExecutionPolicy Bypass
.\start_everydayperfect.ps1
```

停止：

```powershell
.\stop_everydayperfect.ps1
```

### 3.2 手动分开启动

当总启动脚本不方便排查时，建议分开启动。

后端：

```powershell
.\.venv\Scripts\python.exe -m uvicorn app.main:app --app-dir backend --host 127.0.0.1 --port 18765
```

前端：

```powershell
cd frontend
npm.cmd run dev -- --host 127.0.0.1 --port 5173
```

## 5. 清理缓存与本地产物

### 5.1 常规清理

执行：

```powershell
.\clean_local_workspace.ps1
```

作用：

- 清理构建输出
- 清理前端构建缓存
- 清理运行日志
- 清理安装包打包阶段生成的中间文件

不会影响：

- 源码
- 用户发布目录
- 发送压缩包

### 5.2 深度清理

执行：

```powershell
.\clean_local_workspace.ps1 -Deep
```

额外作用：

- 删除 `frontend/node_modules`

适用场景：

- 前端依赖损坏
- `vite`、`esbuild`、`npm` 运行异常
- 想完全重装前端依赖

执行后通常需要重新安装：

```powershell
cd frontend
npm.cmd install
```

## 7. 发版流程

### 7.1 发版前检查

建议先执行：

```powershell
.\.venv\Scripts\python.exe -m pytest backend\tests -q
cd frontend
npm.cmd run test
npm.cmd run build
cd ..
```

### 7.2 生成用户版本

执行：

```powershell
.\publish_windows_release.ps1
```

该脚本会：

- 构建安装包
- 输出用户软件目录
- 生成可发送 zip
- 生成 manifest 清单，便于核对版本、文件大小和 SHA256

当前标准输出位置：

- 用户软件目录：
  `C:\Users\Lenovo\Desktop\EveryDayPerfect\02-user-software\EveryDayPerfect-<版本号>`
- 发送压缩包：
  `C:\Users\Lenovo\Desktop\EveryDayPerfect\03-send-package\EveryDayPerfect-<版本号>-delivery.zip`
- 发版清单：
  `C:\Users\Lenovo\Desktop\EveryDayPerfect\03-send-package\EveryDayPerfect-<版本号>-manifest.json`

## 8. 版本号更新流程

项目版本号统一维护在：

- `app_metadata.py`

不要手工到处搜索替换版本号，统一使用脚本：

```powershell
.\set_project_version.ps1 -Part patch
```

可选用法：

- `.\set_project_version.ps1 -Part minor`
- `.\set_project_version.ps1 -Part major`
- `.\set_project_version.ps1 -Version 1.0.1`

推荐约定：

- 修 bug、小修小补：`patch`
- 有新增功能但兼容旧用法：`minor`
- 有明显不兼容调整：`major`

推荐迭代顺序：

1. 确认本轮要发布的版本号
2. 执行 `.\set_project_version.ps1`
3. 更新必要文档和说明
4. 更新 `CHANGELOG.md`
5. 跑测试与本地验证
6. 执行 `.\publish_windows_release.ps1`

## 9. 目录使用边界

为了避免开发区和交付区混在一起，后续必须遵守：

- 改代码，只在 `01-source\EveryDayPerfect` 操作
- 给用户的软件，只看 `02-user-software`
- 对外发送，只用 `03-send-package`
- 产品材料和说明文档，放 `04-product-docs`

不要在 `02-user-software` 里手改源码，也不要把 `01-source` 直接发给用户。

## 10. 常见 Windows 注意事项

### 9.1 PowerShell 执行策略

如果脚本被拦：

```powershell
Set-ExecutionPolicy -Scope Process -ExecutionPolicy Bypass
```

### 9.2 Node 版本

推荐使用：

```text
Node.js 20 LTS
```

### 9.3 `esbuild` / `spawn EPERM`

如果前端启动时报这类错误，通常不是代码逻辑错，而是 Windows 权限或安全软件拦截。优先检查：

- Node 是否为 20 LTS
- 是否需要管理员 PowerShell
- 是否需要在 Windows 安全中心放行 `esbuild.exe`

### 9.4 `.run` 清不掉

当前 `.run` 目录下可能会遗留两个受保护目录：

- `backend-test-dbs`
- `pytest-temp`

如果 `clean_local_workspace.ps1` 没删掉，通常不是脚本问题，而是 ACL 或占用问题。它们不会影响日常继续开发，可以暂时保留。

如果确实要彻底清掉，建议用管理员 PowerShell 单独处理。

## 11. 建议的迭代节奏

每次迭代建议按这个顺序：

1. 在 `01-source\EveryDayPerfect` 改代码
2. 确认版本号，并执行 `.\set_project_version.ps1`
3. 更新 `CHANGELOG.md`
4. 本地启动并验证核心流程
5. 跑测试
6. 如有需要，执行 `.\clean_local_workspace.ps1`
7. 需要对外版本时执行 `.\publish_windows_release.ps1`

这样可以最大程度避免：

- 源码区和交付区混用
- 构建残留影响下一次开发
- 手工打包导致版本输出不一致
