# 标准开发流程

这份文档是 EveryDayPerfect 后续开发的统一执行标准。  
以后不管是你自己继续做，还是别人接手开发，都优先按这份文档走。

## 1. 先明确工作边界

- 只在 `01-source\EveryDayPerfect` 改源码
- `02-user-software` 只放用户可用的软件版本
- `03-send-package` 只放对外发送的压缩包和 manifest
- `04-product-docs` 只放产品资料和外部说明

不要在发版目录里直接改代码，也不要把源码工程直接发给用户。

## 2. 每次开发前先做什么

开始新一轮开发前，固定先检查：

1. 当前是否在正确目录：`01-source\EveryDayPerfect`
2. 本地 Git 工作区是否干净
3. 当前版本号是什么
4. `CHANGELOG.md` 是否已经记录上一版
5. 当前要做的是：
   - 新功能
   - 修 Bug
   - 文档整理
   - 正式发版
6. 如果 Git 提示 `detected dubious ownership`，先执行：

```powershell
git config --global --add safe.directory C:\Users\Lenovo\Desktop\EveryDayPerfect\01-source\EveryDayPerfect
```

说明：

- 项目内 Git 脚本已经会自动处理这件事
- 只有你手工直接敲 Git 命令时，才可能需要自己补一次

## 3. 日常开发顺序

每次普通开发统一按这个顺序：

1. 拉清楚当前任务目标
2. 改代码
3. 本地启动项目
4. 手工验证关键流程
5. 跑测试
6. 更新相关文档
7. 提交 Git

推荐命令：

```powershell
Set-ExecutionPolicy -Scope Process -ExecutionPolicy Bypass
.\start_everydayperfect.ps1
```

## 4. 改完代码后最低检查项

至少确认下面这些：

- 应用能启动
- 关键页面没有白屏
- 你改动的功能能实际点通
- 没把测试数据带进交付目录
- 没误改 `02-user-software` 和 `03-send-package`

## 5. 什么时候更新版本号

只有在准备正式发版时，才更新版本号。  
平时开发和中间验证，不要频繁改版本号。

规则：

- 修复问题：`patch`
- 兼容性新增功能：`minor`
- 破坏性调整：`major`

示例：

```powershell
.\set_project_version.ps1 -Part patch
```

## 6. 什么时候更新 CHANGELOG

只要准备发版，就必须更新 `CHANGELOG.md`。  
没有版本记录，不允许发给用户。

至少要写清楚：

- 版本号
- 日期
- 新增了什么
- 修复了什么
- 有什么行为变化

## 7. 标准发版流程

正式发版统一按这条路径：

1. 确认代码已稳定
2. 更新版本号
3. 更新 `CHANGELOG.md`
4. 运行测试
5. 构建安装包
6. 生成 `02-user-software`
7. 生成 `03-send-package`
8. Git 提交
9. 打 Git tag
10. 推送 GitHub

推荐直接使用统一脚本：

```powershell
powershell -ExecutionPolicy Bypass -File .\tools\run_standard_release.ps1 -Part patch -Commit -Tag -Push
```

这条脚本会自动检查当前仓库是否需要补 `safe.directory`。

## 8. Git 提交规范

推荐提交信息：

- `feat: ...`
- `fix: ...`
- `docs: ...`
- `chore: ...`
- `release: v1.0.3`

不要使用看不懂的临时提交信息，比如：

- `update`
- `test`
- `again`

## 9. Git 标签规范

正式版本统一使用：

```text
v主版本.次版本.修订号
```

例如：

- `v1.0.2`
- `v1.0.3`

## 10. GitHub 管理规则

源码版本统一在 GitHub 上管理：

- `main`：保持可发布
- 正式版本依赖 Git tag 标识
- GitHub Release 可挂安装包压缩包和 manifest

配套规则详见：

- `docs/GitHub版本管理流程.md`

## 11. 交付物规则

发给用户前必须确认：

- 安装包在 `02-user-software\EveryDayPerfect-<版本号>`
- 发送包在 `03-send-package`
- 包内不含测试数据库、日志、AI 本地配置
- 有 `README-user.txt`
- 有 manifest

## 12. 接手开发的人先看哪些文件

固定顺序：

1. `README.md`
2. `docs/开发交接手册.md`
3. `docs/标准开发流程.md`
4. `docs/GitHub版本管理流程.md`
5. `docs/开发维护文档.md`
6. `CHANGELOG.md`

## 13. 一句话执行标准

以后 EveryDayPerfect 的开发、发版、交付，统一遵循：

```text
先在源码目录开发 -> 本地验证 -> 更新文档 -> 标准发版 -> Git 提交与打标签 -> GitHub 留痕 -> 交付用户
```
