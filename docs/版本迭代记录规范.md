# 版本迭代记录规范（简中）

## 目标
- 每次版本更新都可追溯：做了什么、为什么做、怎么验证、如何回滚。
- 让开发、测试、交付、运维在交接时有统一口径。

## 必须更新的文件
- `CHANGELOG.md`：记录用户可见变化（功能、修复、行为变化）。
- `docs/开发维护文档.md`：更新运行/排障/注意事项（如有变化）。
- `docs/开发交接手册.md`：更新流程或职责（如有变化）。

## 每次发版最少记录内容
1. 版本号与日期
2. 本次目标（1-2 句话）
3. 功能新增
4. 问题修复
5. 兼容性影响（是否影响旧数据/旧配置）
6. 验证结果（测试命令 + 结果）
7. 交付物路径（安装包、压缩包）
8. 回滚方案（回滚到哪个版本，如何操作）

## 版本条目模板
将以下模板复制到 `CHANGELOG.md`：

```md
## x.y.z - YYYY-MM-DD

- 目标：一句话说明本次迭代目标。
- 新增：
  - ...
- 修复：
  - ...
- 调整：
  - ...
- 风险与兼容性：
  - ...
- 验证：
  - 前端：`npm run build`（通过/失败）
  - 后端：`pytest ...`（通过/失败）
  - 安装包：`build_windows_installer.ps1`（通过/失败）
- 交付物：
  - 安装包：`dist/installer/xxx.exe`
  - 发布压缩包：`dist/release/xxx.zip`
- 回滚：
  - 回滚版本：...
  - 操作说明：...
```

## 命名建议
- 版本号采用语义化：`主版本.次版本.修订号`（如 `1.0.1`）。
- 仅修复问题：提升修订号（`x.y.+1`）。
- 新增功能但兼容：提升次版本号（`x.+1.0`）。
- 有破坏性变更：提升主版本号（`+1.0.0`）。

## 执行规则（必须）
- 没有更新 `CHANGELOG.md` 的代码，不允许发版。
- 没有“验证结果”的版本记录，不允许发给用户。
- 所有记录统一使用简体中文，避免乱码。
