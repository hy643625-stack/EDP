# 开发交接手册

这份文档是给“接手这个项目继续开发的人”用的。目标不是讲所有背景，而是让接手者在最短时间内知道：

- 代码在哪里改
- 项目怎么启动
- 版本怎么更新
- 发版怎么做
- 遇到常见问题先查什么

## 1. 先记住这 4 个目录

整个桌面工作区分成四块：

- `01-source`：源码开发区
- `02-user-software`：面向用户的软件目录
- `03-send-package`：对外发送的压缩包
- `04-product-docs`：产品资料和说明文档

真正开发只在这里进行：

```text
C:\Users\Lenovo\Desktop\EveryDayPerfect\01-source\EveryDayPerfect
```

不要在下面两个目录里改代码：

- `02-user-software`
- `03-send-package`

它们是发版输出区，不是开发区。

## 2. 第一次接手时先做什么

### 2.1 打开这几个文件

优先看这几个文件：

- `README.md`
- `docs/开发交接手册.md`
- `docs/开发维护文档.md`
- `docs/标准开发流程.md`
- `docs/GitHub版本管理流程.md`
- `CHANGELOG.md`

### 2.2 确认环境

建议基线：

- Windows
- Node.js 20 LTS
- Python 3.12+ 或当前已可用的 Python 3 环境

### 2.3 启动项目

在源码工程根目录执行：

```powershell
Set-ExecutionPolicy -Scope Process -ExecutionPolicy Bypass
.\start_everydayperfect.ps1
```

默认地址：

- 前端：`http://127.0.0.1:5173`
- 后端：`http://127.0.0.1:18765`

停止服务：

```powershell
.\stop_everydayperfect.ps1
```

## 3. 平时开发怎么做

建议固定按下面顺序：

1. 在 `01-source\EveryDayPerfect` 改代码
2. 本地启动并验证修改
3. 跑测试
4. 更新版本号
5. 更新 `CHANGELOG.md`
6. 如有需要清理本地产物
7. 需要对外版本时再导出发版

最重要的一条规则：

- 没准备发版时，不要动 `02-user-software` 和 `03-send-package`

## 4. 版本号怎么更新

项目版本号统一维护在：

- `app_metadata.py`

不要手工在多个文件里搜版本号。统一用脚本：

```powershell
.\set_project_version.ps1 -Part patch
```

常用方式：

- 修 bug：`.\set_project_version.ps1 -Part patch`
- 增加兼容功能：`.\set_project_version.ps1 -Part minor`
- 有不兼容改动：`.\set_project_version.ps1 -Part major`
- 明确指定版本：`.\set_project_version.ps1 -Version 1.0.1`

## 5. 测试和本地验证

推荐发版前至少执行：

```powershell
.\.venv\Scripts\python.exe -m pytest backend\tests -q
cd frontend
npm.cmd run test
npm.cmd run build
cd ..
```

如果只是开发中验证，也至少要做这两件事：

- 核心页面能打开
- 核心流程能走通：创建任务、查看项目管理、创建待办、保存记录

## 6. 本地缓存和产物怎么清理

常规清理：

```powershell
.\clean_local_workspace.ps1
```

这个脚本会清理：

- `build/`
- `dist/`
- `frontend/dist/`
- `frontend/*.tsbuildinfo`
- `.run` 里的日志和 PID
- 打包生成的中间文件

深度清理：

```powershell
.\clean_local_workspace.ps1 -Deep
```

这个会额外删除：

- `frontend/node_modules`

什么时候用深度清理：

- 前端依赖坏了
- `vite` / `esbuild` / `npm` 出现异常
- 想完整重装前端依赖

## 7. 发版怎么做

导出发版统一用：

```powershell
.\publish_windows_release.ps1
```

这个脚本会自动完成：

- 构建安装包
- 刷新 `02-user-software` 下对应版本目录
- 生成 `03-send-package` 下的发送 zip
- 生成对应版本的 manifest 清单

标准输出位置：

- 用户软件目录：
  `C:\Users\Lenovo\Desktop\EveryDayPerfect\02-user-software\EveryDayPerfect-<版本号>`
- 发送压缩包：
  `C:\Users\Lenovo\Desktop\EveryDayPerfect\03-send-package\EveryDayPerfect-<版本号>-delivery.zip`
- 发版清单：
  `C:\Users\Lenovo\Desktop\EveryDayPerfect\03-send-package\EveryDayPerfect-<版本号>-manifest.json`

## 8. 交接前必须确认什么

如果你准备把当前迭代交给下一个人，至少确认下面这些：

1. 当前代码已经放在 `01-source\EveryDayPerfect`
2. 当前版本号已经更新
3. `CHANGELOG.md` 已更新
4. 文档里提到的启动方式仍然可用
5. 测试是否通过，有没有跳过项，已经写清楚
6. 如果导出了版本，`02-user-software` 和 `03-send-package` 已经刷新
7. 如果有已知问题，写进 `CHANGELOG.md` 或维护文档，不要只留在口头说明里

## 9. 遇到问题先看哪里

优先检查：

- `README.md`
- `docs/开发维护文档.md`
- `docs/故障排查.md`
- `.run/logs/`

常见问题关键词：

- PowerShell 脚本被拦
- 数据库只读
- 前端连接不到后端
- `esbuild` / `spawn EPERM`
- `.run` 目录删不掉

## 10. 现在这套流程的核心原则

为了后续交接不混乱，默认遵守这三条：

1. 开发只在 `01-source`
2. 交付只看 `02-user-software` 和 `03-send-package`
3. 版本更新、清理、发版尽量都走脚本，不走手工零散操作
