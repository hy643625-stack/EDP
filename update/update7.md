一、 属性输入类型精简与卡片表层 UI (Surface UI) 重构

数据类型归拢： 后端/前端的数据模型中，属性的 input_type 仅保留三种核心形态：boolean (打卡型), number (数值型), timer (计时型)，创建任务属性时默认类型给到 number，用户可选

卡片右侧快捷操作区渲染规则：

boolean：渲染为一个圆形大 Checkbox，点击直接完成。

number：渲染为一个轻量的组合：左侧是极简的文字输入框（展示当前进度/目标），右侧紧跟一个 [ +1 ] 的快捷按钮。点击 +1 直接触发乐观更新。

timer：渲染为一个 [ ▶️ 开始专注 ] 按钮。

二、 引入“任务级”沉浸式记录面板 (Task-Level Focus Modal)

触发机制： 用户点击某张 number 卡片的输入框区域，或点击 timer 卡片的开始按钮，不要只针对单一属性弹窗，而是唤起一个属于该卡片母任务（Task）的底部抽屉 (Bottom Sheet) —— 「 任务专注空间 (Task Focus Mode) 」。

面板数据注入： 当抽屉弹出时，前端需要将该 Task 下的所有今日活跃属性一起带入面板。

面板布局结构：

顶部 Header： 显示 Task 名称和颜色 Badge，带有 [ 关闭/收起 ] 按钮。

时钟区 (核心)： 如果触发动作是点击 timer，则顶部直接展开一个大号的正在运行的计时器（正计时/倒计时）。关键逻辑： 该计时器结束后的时长，不仅记录为该 timer 属性的进度，必须同步计入该 Task 的 专注时长（默认属性） 中。

属性聚合区： 在时钟下方，纵向列出该 Task 下所有的 number 和 boolean 属性。这里使用原生的大号 Input 框和明显的打卡按钮，方便用户在“专注期间”或“专注结束”时，一次性把所有相关数据填完。

重塑带“超额完成”特效的卡片进度条 (Progress Bar & Over-achievement)
一、 核心诉求 (Objective)
在聚合页的属性卡片中，恢复直观的进度条反馈机制。必须支持“正常进度”和“超额完成 (Over-achievement)”两种视觉状态，保留用户填满进度条时的“多巴胺爽感”。

二、 UI 视觉规范 (Visual Specs)

布局位置 (Layout)：
将进度条放置在卡片的偏下方（在属性名和数值的下方，快捷操作按钮的左侧或贯穿整个卡片底部）。
不要让它像以前那样占据太大的纵向空间，可以设计成高度约 6px - 8px 的圆角进度条 (Rounded Track)。

数据展示层 (Data Label)：
在进度条的上方或内部，用极简的排版显示当前值与目标值。
例如：用加粗的大字体显示当前值，小字体灰色显示 / 目标值（如：80 / 100）。

双重进度条逻辑 (The Dual-Track Logic)：
这是整个设计的灵魂。前端需要根据 (current_value / target_value) * 100 计算百分比，并渲染两种状态：

状态 A：正常进度 (0% - 100%)

基础底槽：浅灰色或极淡的品牌色背景。

填充条：使用该属性所属 Task 的专属颜色（Task Color），或者系统统一的品牌主色（如紫色）。

状态 B：超额完成 (> 100%)

底层： 原本的 100% 进度条保持填满状态，颜色可以稍微加深或变为稳定的“完成态”颜色。

高亮覆写层 (Over-fill Layer)： 从 0% 开始，叠加一层全新的、极具成就感的颜色（比如渐变金、荧光绿、或带有流光动画的亮色）。

计算公式： 超额百分比 = ((current_value - target_value) / target_value) * 100。这层高亮条就按照这个新的百分比在原有的进度条上“再跑一圈”。

三、 交互融合 (Interaction)
虽然有了进度条，但依然要保持我们之前定的“轻量化操作”。
在进度条的右侧，保留那个小巧的 [ +1 ] 按钮。用户每点一次，进度条就丝滑地向前推进一截（必须带有平滑的 CSS Transition 动画，如 transition: width 0.3s ease-out），这种“指尖上的即时反馈”是绝佳的体验。
