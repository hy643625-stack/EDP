# 📝 架构与实现文档：任务属性的“高维周期与目标共享”机制

## 1. 业务背景与目标
现有的“任务属性（Attribute）”为简单的单日无限期循环为了适应真实场景中弹性的精力分配，并为未来的 AI Agent 介入打下结构化基础，我们需要引入**“高维周期管理”**
系统需支持两种维度的属性设定：
* **基础模式（默认）：** 每日生效，单日独立结算（目前的任务属性都是这个模式）
* **高级模式（弹性/共享）：** 指定生效的星期数，并支持跨日期的“目标共享”（如：周六、周日两天共享“阅读 50 页”的总目标）

## 2. 核心数据结构升级 (Data Schema)
**必须严格遵循声明式数据结构** 无论用户在前端如何复杂地操作，最终落库或传递给 AI Agent 的数据格式必须如下标准化

请在原有的 `Attribute` 模型中，扩充或重构周期调度相关的字段 `schedule_config`
例如：（是例如！原有结构要保留！新结构要拓展！）
```json
{
  "attribute_id": "attr_10293",
  "name": "周末深度阅读",
  "target_value": 50,
  "unit": "页",
  "schedule_config": {
    "type": "shared_days",       // 核心字段！可选值: "daily" (每日), "specific_days" (指定日独立结算), "shared_days" (指定日共享目标)
    "active_weekdays": [6, 7],   // 仅在周六、周日生效 (1=周一, 7=周日)
    "period_start": "2026-04-16",// 周期开始时间 (留空为无限期)
    "period_end": "2026-05-16"// 周期结束时间 (留空为无限期)
  }
}
```

## 3. 核心业务逻辑实现 (Business Logic)

### 3.1 渲染过滤逻辑（Records 页面今日清单）
当渲染今日打卡列表时，前端/逻辑层需进行判断：
1. 检查 `schedule_config.active_weekdays` 是否包含当前日期的星期值如果不包含，**不要在今日清单中渲染该属性**
2. 检查当前日期是否在 `period_start` 和 `period_end` 的范围内如果过期，挂起该属性并触发“复盘工作流”（Phase 2 实现，本次先做过滤）

### 3.2 进度计算逻辑（共享目标的难点）
当 `schedule_config.type === "shared_days"` 时，进度的计算方式必须改变：
* **常规状态 (daily/specific_days)：** `进度 = 今日录入值 / target_value`
* **共享状态 (shared_days)：** 需要向前/向后追溯当前循环周期（如本周的周六到周日）`进度 = (本周六录入值 + 本周日录入值) / target_value`
* **UI 呈现：** 如果今天是周六，用户录入了 20 页，UI 应显示进度为 `20/50`到了周日，起步进度依然是 `20/50`，用户再录入 30 页，则本周期的该属性达成 100%

## 4. 前端 UI/UX 重构指令 (Progressive Disclosure)

### 4.1 属性创建/编辑表单
必须采用**“渐进式明细 (Progressive Disclosure)”**设计，严禁将复杂表单直接平铺
* **默认视图：** 只展示名称、目标值、单位隐藏所有周期设置，默认按 `"type": "daily"` 处理
* **高级开关：** 在表单底部提供一个次级按钮或折叠面板：`[ ⚙️ 展开高级周期设置 ]`
* **高级视图展开后：**
    * 渲染7个格子代表一周的7天，点击第i个代表任务在周i并以天为单位生效，如：点击1，3代表周1和周3要进行这个任务
    * 划线选择多天代表这些天要`[ 开启多日目标共享 ]`，并以这些天为单位共享目标，如：鼠标划过6、7格子，该任务属性还在[6,7]生效并且共享目标
    * 对于每个属性选定的小周期可以设置相同的目标，也可以设置独立的目标值，如：勾选周1、3并且划选6、7，其中6、7开启了多日目标共享，那么我就可以为[1]\[3]\[6,7]独立设置目标值

### 4.2 Records 页面卡片展示优化
对于设定了“共享目标”的属性，卡片上的提示语需要调整，避免用户困惑：
* 在属性名旁增加一个小 Badge 或 Icon（如 🔗），Hover 时提示：“该目标由周六、周日共享”

## 5. 开发步骤建议
1.  **Step 1:** 先更新底层 Store / 数据库的 Schema，使其支持 `schedule_config`
2.  **Step 2:** 修改前端表单，实现带有“高级选项”折叠的创建/编辑交互，确保能正确生成规范的 JSON 数据
3.  **Step 3:** 重写 Records 页面的过滤和计算逻辑，使其能够正确解析 `specific_days` 和 `shared_days` 的展示与累加规则
